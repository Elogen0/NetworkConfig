쿠버네티스(k8s)와 AWS 환경에서 **'노드(Node)'**와 **'VM(가상 머신)'**은 실무적으로 거의 동일하게 취급되지만, 기술적으로는 **'역할'**과 **'실체'**라는 관점에서 차이가 있습니다.

# Node란
Node는 쿠버네티스 클러스터 안에서 실제로 Pod가 실행되는 머신


### Node의 구성요소
- kubelet 
노드의 관리자<br>
“이 노드에서 어떤 Pod를 실행해라”라는 지시를 받아서 실행함<br>
Pod 상태를 Control Plane에 계속 보고<br>

- Container Runtime
실제로 컨테이너를 실행하는 엔진<br>
예: containerd, CRI-O (예전엔 Docker도 많이 씀)

- kube-proxy
네트워크 담당<br>
서비스(Service) → Pod 트래픽 라우팅 처리

### 마스터 노드와 워커 노드
* **마스터 노드 (Control Plane):** 클러스터를 관리하는 뇌 역할을 합니다. EKS에서는 AWS가 이를 관리하므로 사용자는 VM 목록에서 마스터 노드를 볼 수 없습니다.
* **워커 노드 (Worker Node):** 실제 파드가 뜨는 곳입니다. 사용자의 AWS 계정에 **EC2 인스턴스** 형태로 나타나며, 이것이 우리가 흔히 말하는 'VM'입니다. 보통은 Pod를 안 올리게 taint 걸어둠
```
[ Control Plane ]
- API Server
- Scheduler
- Controller Manager
- etcd

[ Node ]
- kubelet
- container runtime
- kube-proxy
- Pods
```

## Node의 역할
- Pod를 실행한다
- 리소스(CPU, 메모리)를 제공한다
- 네트워크와 스토리지를 연결한다
- 상태를 Control Plane에 보고한다

# Node와 vm의 차이
### 1. 역할과 실체의 차이

* **가상 머신 (VM):** AWS에서는 이를 **EC2 인스턴스**라고 부릅니다. 하이퍼바이저 위에서 작동하는 하나의 독립된 컴퓨팅 자원(실체)을 의미합니다.
* **노드 (Node):** 쿠버네티스 클러스터의 일원이 되어 **컨테이너를 실행할 수 있는 상태의 서버**를 말합니다.

즉, **"AWS EC2(VM) 위에 쿠버네티스 에이전트(kubelet)를 설치하면, 그때부터 그 EC2는 노드가 된다"**고 이해하는 것이 가장 정확합니다.

### 2. AWS에서의 예외 케이스 (Fargate)

AWS EKS를 사용하면 **"노드 = VM" 공식이 깨지는 경우**가 있습니다. 바로 **AWS Fargate**를 사용할 때입니다.

| 구분 | 일반적인 노드 (EC2 기반) | 서버리스 노드 (Fargate 기반) |
| --- | --- | --- |
| **실체** | 사용자가 관리하는 EC2 인스턴스 (VM) | AWS가 관리하는 추상화된 실행 환경 |
| **VM 접근** | SSH 등으로 직접 접속 가능 | 서버 자체에 접속 불가 (파드만 실행) |
| **관리 주체** | 사용자가 OS 패치 및 런타임 관리 | AWS가 하부 인프라 전체 관리 |

이 경우, 논리적으로는 '노드'라는 단위로 취급되지만 사용자가 관리하는 'VM'은 존재하지 않게 됩니다.

### 3. 실무 및 과금 주의사항

* **인스턴스 타입 선택:** 노드(VM)의 사양(vCPU, RAM)에 따라 파드를 배치할 수 있는 개수가 결정됩니다. 너무 큰 VM은 자원 낭비를, 너무 작은 VM은 빈번한 스케줄링 실패를 야기합니다.
* **유료 서비스 주의:** EC2 기반 노드를 사용하면 **인스턴스 시간당 비용**이 발생하며, 별도로 **EKS 클러스터 관리 비용(시간당 $0.10)**이 부과됩니다. 노드 개수가 늘어나거나 고사양 EC2(m5.large 이상 등)를 사용하면 비용이 급격히 증가할 수 있으니 주의가 필요합니다.
